# Receipt24: Phase 1–2 foundation

This repository currently contains only the platform foundation. It deliberately has no consumer dashboard, accountant portal, administrator dashboard, merchant workflow, OCR integration, payment integration, or frontend application.

## Architecture and environments

The planned topology is a shared Supabase project per environment:

| Environment | Purpose | Supabase project | App configuration |
| --- | --- | --- | --- |
| Development | Local developer work | Supabase CLI local stack | `.env.local` (not committed) |
| Testing | Automated/integration testing | Separate non-production project | CI secrets |
| Production | Customer data | Dedicated production project | Deployment secrets |

`auth.users` is the identity store; `public.users` is the application profile. The `handle_new_auth_user` trigger provisions a consumer profile row when Supabase Auth creates an identity. A trusted server-side onboarding flow must create accountant roles after it validates an application; public sign-up cannot assign a role.

Only these roles exist: `consumer`, `accountant`, `accounting_firm_manager`, `super_administrator`, and `support_administrator`. There are no merchant roles, accounts, subscriptions, permissions, or dashboards. `merchants` is receipt-reference data only.

Tables:

| Area | Tables |
| --- | --- |
| Identity | `users`, `consumer_profiles`, `accountants`, `accounting_firm_members` |
| Receipt data | `merchants`, `receipts`, `receipt_items`, `receipt_uploads` |
| Classification | `receipt_categories`, `expense_categories`, `receipt_expense_classifications` |
| Lifecycle | `warranties`, `returns_and_refunds`, `accountant_client_access` |
| Operations | `notifications`, `subscriptions`, `support_tickets`, `audit_logs` |

All requested fields, constraints, creation/update timestamps, indexes, enums, foreign keys, and `updated_at` triggers are defined by the first migration. The second migration owns security functions, RLS policies, and storage buckets.

## Relationships

```text
auth.users 1—1 users
users 1—0..1 consumer_profiles
users 1—0..1 accountants 1—N accounting_firm_members
users 1—N receipts N—1 merchants
receipts 1—N receipt_items, receipt_uploads, warranties, returns_and_refunds
receipts 1—0..1 receipt_expense_classifications N—1 expense_categories
receipts N—1 receipt_categories
accountants N—N users (consumers) through accountant_client_access
users 1—N notifications, support_tickets, audit_logs, subscriptions
```

## RLS policy model

RLS is enabled on every application table. Policies are deny-by-default:

- Consumers can create, read, and update their own receipts and dependent records. Receipt deletion is intentionally unavailable.
- Accountants can only read receipt data after an active, approved client grant matches its scoped date range and scope type (`all_receipts`, `business_only`, `tax_related`, or `selected_categories`).
- Consumers own access-grant approvals and revocations. Approved accountants create invitations only.
- Category vocabularies are readable by authenticated users and writable only by active platform administrators.
- An accountant can view and edit their pending verification application but cannot self-approve it.
- Application role, account status, and verification flags are protected from self-escalation by a trigger.
- Administrators receive no broad receipt-table bypass. Support, fraud, legal, and security access must be implemented as service-role Edge Functions that create an `audit_logs` record and enforce an approved case reason before returning sensitive data.
- User-facing subscription, ticket, notification, and audit-log access is limited to the record owner.

## Storage

| Bucket | Visibility | Allowed MIME types | Object path |
| --- | --- | --- | --- |
| `receipt-files` | Private | JPEG, PNG, HEIC, PDF | `<user-id>/<opaque-file-name>` |
| `accountant-verification` | Private | JPEG, PNG, PDF | `<user-id>/<opaque-file-name>` |

Storage policies require the first path segment to match `auth.uid()`. Receipt sharing uses signed URLs generated by a future audited Edge Function; it does not make a bucket public.

## Configuration required before Phase 3

1. Create three Supabase projects and link the local CLI to development.
2. Set all secrets in `.env.example` through the deployment secret manager; never commit real values.
3. Configure Supabase Auth providers, email verification, redirect URLs, password policy, MFA, and rate limits.
4. Configure OCR, email ingestion, push notification, payment, malware-scanning, analytics, and signed-URL Edge Functions.
5. Define administrator case-review and audit workflow for sensitive support access.
6. Confirm retention, backups, incident response, POPIA/GDPR review, and legal copy with qualified professionals.
7. Add country, currency, language, category, and subscription-plan seed data through a controlled migration.

## Phase 1–2 testing checklist

- [ ] `supabase db reset` applies both migrations to a clean local database.
- [ ] Auth sign-up creates a `users` record with `consumer` role only.
- [ ] A client cannot alter its role, account status, or verification flags.
- [ ] A consumer can only read/write its own receipt hierarchy.
- [ ] A second consumer cannot query the first consumer's receipts, uploads, warranties, returns, or classifications.
- [ ] An accountant has no receipt access before consumer approval.
- [ ] An approved accountant receives only the records allowed by its scope; revocation immediately removes access.
- [ ] A user cannot access another user's objects in either private storage bucket.
- [ ] Neither `anon` nor `authenticated` clients can directly bypass the receipt policies.
- [ ] Service-role workflows write a corresponding `audit_logs` event for privileged access.

## Known limitations / unresolved work

- No Supabase project has been linked, so migrations have not been executed against a real database.
- The local Supabase CLI/Docker test stack is required to execute RLS integration tests.
- Admin support access, role provisioning, audit log writing, storage signed URLs, and accountant invitation delivery require Edge Functions in a later phase.
- The specified email-import address, OCR workflows, notifications, payment provider, and external accounting exports are intentionally not implemented in this phase.
